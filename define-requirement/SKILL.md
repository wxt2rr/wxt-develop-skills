# 明确需求（把话说死）

用于把模糊想法、半成品指令、带方案的需求描述，收敛成可继续设计和开发的明确边界。

## 目标

1. 把“想做什么”与“打算怎么做”拆开。  
2. 写清目标、范围、非目标、约束、角色、输入输出和成功标准。  
3. 让下一阶段不再猜用户真正要什么。  

## 适用场景

1. 用户描述偏口语、跳步、夹带实现方案。  
2. 需求里同时混着目标、限制、猜想和抱怨。  
3. UI 行为、接口增强、流程调整、产品方案需要先锁边界。  

## 执行流程

1. 复述当前理解  
   - 用更短、更准的语言复述用户要的结果。  
   - 发现歧义时先指出歧义，不抢着拍板。  
2. 拆目标与方案  
   - 把“目标行为”和“用户顺手提出的实现方式”分开。  
   - 若实现方式不是硬约束，不要提前绑定。  
3. 写边界  
   - 写清范围、非目标、兼容要求、风险约束。  
   - 标出哪些内容本次不做。  
4. 写验收  
   - 至少覆盖主路径、关键边界、失败路径或异常展示。  
   - 验收标准必须能被后续验证。  
5. 标注待确认项  
   - 只保留真正影响方案或实现的待确认问题。  
   - 不为了显得谨慎而堆问题。  

## 输出模板

1. 当前理解  
2. 目标  
3. 范围  
4. 非目标  
5. 约束与兼容要求  
6. 验收标准  
7. 待确认项  

## 质量闸门

1. 没有非目标，就容易超范围。  
2. 没有成功标准，就不能进入方案阶段。  
3. 待确认项只保留关键分歧，不能把“还没想”包装成“已澄清”。  
4. 需求足够明确后，再切到 `design-solution`。
