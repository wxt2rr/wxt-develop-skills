# 明确需求（把话说死）

用于把模糊想法、半成品指令、带方案的需求描述，结合当前项目上下文，收敛成用户明确认可的需求边界。

## 目标

1. 把“想做什么”与“打算怎么做”拆开。  
2. 先把需求绑定到当前对话所在项目，再判断改动归属、范围和成功标准。  
3. 发现阻塞分歧时立即发问，一次只问一个，不把问题堆到最后。  
4. 主动识别用户没提到、但很可能出问题的隐含风险点。  
5. 在用户明确认可需求定义前，不进入 `design-solution`。  

## 适用场景

1. 用户描述偏口语、跳步、夹带实现方案。  
2. 需求里同时混着目标、限制、猜想和抱怨。  
3. UI 行为、接口增强、流程调整、产品方案需要先锁边界。  
4. 当前对话已经落在某个具体项目中，需要先判断这次改动究竟属于前端、后端、双端联动，还是超出当前项目承接范围。  

## 执行流程

1. 绑定当前项目上下文  
   - 默认把需求放到当前对话所在 workspace 中理解。  
   - 先根据当前项目的目录、入口、模块职责、已有接口和页面位置，判断这次改动归属：`前端 / 后端 / 前后端联动 / 当前项目无法承接 / 需跨项目`。  
   - 能从当前项目直接推断的，不回抛给用户确认。  
2. 形成当前理解  
   - 用更短、更准的语言复述用户要的结果。  
   - 发现歧义时先指出歧义，不抢着拍板。  
3. 即时发问澄清  
   - 只要发现一个会改变方向、范围或验收的阻塞分歧，就立刻发问。  
   - 一次只问一个问题，等用户回答后再继续。  
   - 能从当前项目推断的，不提问。  
4. 拆目标与方案  
   - 把“目标行为”和“用户顺手提出的实现方式”分开。  
   - 若实现方式不是硬约束，不要提前绑定。  
5. 写边界  
   - 写清范围、非目标、兼容要求、风险约束。  
   - 标出哪些内容本次不做。  
6. 写验收  
   - 至少覆盖主路径、关键边界、失败路径或异常展示。  
   - 验收标准必须能被后续验证。  
7. 识别隐含风险  
   - 主动检查用户没提到、但后续高概率会出问题的点，例如：改动归属是否错位、兼容旧行为、权限边界、历史数据影响、异步链路影响、前后端口径不一致。  
   - 只有当这些风险会改变方向、范围或验收时，才升级为立即确认项。  
8. 输出需求定义并请求确认  
   - 在阻塞分歧被逐个消化后，输出一版完整需求定义。  
   - 明确请求用户确认这版需求定义是否成立。  
   - 未经确认，不进入 `design-solution`。  

## 输出模板

1. 当前理解  
2. 项目归属与改动面判断  
3. 目标  
4. 范围  
5. 非目标  
6. 约束与兼容要求  
7. 验收标准  
8. 已识别风险点  
9. 当前需要你确认的唯一问题（若无则省略）  

## 提问规则

1. 先看当前项目，再决定要不要问。  
2. 一次只问一个问题。  
3. 只问会改变方向、范围、验收或改动归属的问题。  
4. 不把多个问题攒到最后统一抛给用户。  
5. 用户回答后，先更新当前理解，再决定是否还要继续问下一个。  

## 质量闸门

1. 没有非目标，就容易超范围。  
2. 没有成功标准，就不能进入方案阶段。  
3. 没结合当前项目上下文，就不能判断改动归属。  
4. 能从当前项目推断的内容，不得回抛给用户确认。  
5. 发现阻塞分歧时，必须即时提问，不能拖到最后统一列清单。  
6. 如果已识别到高风险隐含问题，必须明确提出，不能假装不存在。  
7. 未经用户明确认可需求定义，不得进入 `design-solution`。
