# 设计方案并锁契约（给唯一推荐）

用于在需求已清楚的前提下，给出单一推荐方案，并把接口、参数、行为契约和验证点一起锁死。

## 目标

1. 只给一个最终推荐方案，不做发散堆砌。  
2. 关键设计结论必须有依据和取舍理由。  
3. 在进入实现前锁定接口、参数、状态流转、兼容和验证方式。  

## 适用场景

1. 需求已经清楚，但还没决定怎么实现最稳。  
2. 需要在新增 / 替换 / 兼容 / 重构之间做取舍。  
3. 需要提前锁接口、字段、事件、页面行为或模块职责。  

## 执行流程

1. 识别事实与约束  
   - 先列出现有代码、已有接口、历史行为、外部依赖。  
   - 把确定事实、推断和设计选择分开。  
2. 给唯一推荐方案  
   - 只保留一个最终方案。  
   - 简述不选其他路线的关键原因。  
3. 锁契约  
   - 写清接口、参数、返回、状态流转、错误处理、兼容规则。  
   - 前端任务补页面行为、交互触发条件和异常表现。  
4. 拆最小实施切片  
   - 每步都要能独立验证。  
   - 每步都要知道影响范围和回滚点。  
5. 写验证矩阵  
   - 明确每个关键改动用什么命令或行为检查来验。  

## 输出模板

1. 当前事实  
2. 关键假设  
3. 方案结论  
4. 不选其他路线的原因  
5. 契约定义  
6. 实施切片  
7. 验证矩阵  
8. 风险与回退  

## 质量闸门

1. 没有事实依据，不拍方案。  
2. 没锁契约，不进入实现。  
3. 没验证路径的方案，视为未完成。  
4. 如果方案需要大改现有边界，先说明兼容与回退，再进入 `implement-change`。
